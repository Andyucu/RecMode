Modern Windows 11 screen recorder (Bandicam-class). **.NET 10 · WPF · Fluent design · MVVM.**
**Win11 is primary; Win10 2004+ must work. Portable-first: ships as a self-contained folder/zip.**

## Key decisions (details + rationale in the plan §3)
- **Resource efficiency is a core product value (§3.9 — binding rules, violations are bugs):** nothing runs unless visible or recording (capture/audio/timers torn down on nav-away/minimize/tray); event-driven never polled; allocation-free hot paths (texture ring, ArrayPool, profiler-verified); one GPU copy chain (NV12, no BGRA round-trips); UI updates throttled+batched (meters ≤30 Hz, stats ≤4 Hz, preview ≤30 fps default, occlusion-paused); hw encoders default; virtualized/cached library; ReadyToRun. Measured budgets (plan §1): tray idle <0.5% CPU / <180 MB; Record screen open <3% CPU; recording 1440p60 hw = app <15% of one core excl. ffmpeg, memory flat over 2 h. Soak harness regression-checks these. **One documented exception to "event-driven never polled" (found by a 2026-07-07 code-health review, kept intentionally rather than "fixed"):** `SchedulerService` polls every ~20s for the app's whole lifetime instead of arming a precise single-shot timer, because schedules match at minute granularity and a `Background`-priority tick could be delayed past its target minute under UI-thread load — polling ~3×/minute is deliberate fault tolerance (a missed tick within the same minute is still caught by the next one), not an oversight. Cost is negligible (one wakeup/20s idle-in-tray). See the class doc comment on `SchedulerService` for the full reasoning.
- **Portable-first (§3.5):** self-contained win-x64 publish; `portable.marker` + `AppPaths` service; all state in `.\Data\`; ffmpeg bundled at `.\ffmpeg\`; recordings default `.\Recordings`; nothing written outside the folder (opt-in exceptions labeled). Installer/auto-update (Velopack) is a later build sharing the same code.
- Capture: Windows.Graphics.Capture + D3D11 (Vortice); region = GPU crop; preview via D3DImage.
- Encoding: ffmpeg.exe subprocess over named pipes, **two-tier strategy with a measured decision gate (§3.3)** — Tier 1 = GPU NV12 convert → readback → pipe (MVP); Tier 2 = in-proc hw frames only if the gate fails. Encoders NVENC/AMF/QSV/x264/x265/SVT-AV1 probed at startup; containers MP4/MKV/MOV/WebM; quality slider → CRF = 51 − q·0.38. Encoder currency (plan §3.3): SVT-AV1 **screen-content mode ON for screen/window/region sources** (palette + intra-block-copy = sharper text/UI); **Opus is first-class audio** (default MKV/WebM; AAC = native ffmpeg encoder, default MP4/MOV since libfdk_aac isn't redistributable); ffmpeg re-pinned to latest stable at least quarterly with snapshot/ffprobe tests guarding drift; VVC + VP9 deliberately excluded.
- **Resource controls (§3.3):** CPU thread cap + effort presets (sw), effort tiers (NVENC p1–p7 / AMF / QSV), encoder process priority — bounded by hardware-derived recommendations; GPU % can't be hard-capped, effort tier is the honest lever.
- Audio: NAudio WASAPI (system loopback + mic) + process-loopback interop for per-app audio (post-MVP); 48 kHz f32 mixer, per-source gain/mute/meters; AAC/FLAC/Opus.
- MVVM: CommunityToolkit.Mvvm; DI: Microsoft.Extensions.Hosting; logging: Serilog; tray: H.NotifyIcon.
- Error taxonomy in Core from day one (§3.6): RecoverableWarning / BlockingError / DegradedState / FatalFinalizationError.
- Theming: port `_ds/tokens/*.css` to Light/Dark dictionaries + 5 accent presets; Mica on Win11 (opaque fallback Win10); 4px grid, 32px controls; Fluent System Icons as XAML geometries; sentence case, `×`, no emoji.
- **Robustness/product pass (plan v2 final):** CFR output policy (duplicate/drop vs QPC clock; VFR post-1.0); **HDR tone-map to SDR inside the NV12 pass** (washed-out-recording bug prevented, HDR passthrough post-1.0); global crash handler + local minidumps; black-frame watchdog for exclusive-fullscreen games; auto-split (FAT32 4 GB aware); CLI + single-instance forwarding (`--record/--screenshot/--stop/--tray`); all strings in .resx from Phase 1; **no telemetry, ever** (privacy is a feature); AMF/QSV beta-labeled until tested on real AMD/Intel hardware; dogfooding from M3; `DOCS/DESIGN_NOTES.md` logs deviations from the design file (starts Phase 6); **post-1.0 backlog is plan §7** (replay buffer ranked #1) — nothing from it creeps into 1.0 **except by explicit user request** (recording profiles/presets, §7 #4, pulled forward 2026-07-06 in scoped-down form — see status tracker).
- **Feature-list triage (2026-07-03, user's idea list):** promoted into 1.0 — safe-recording mode (MKV + auto-remux to MP4, default on, Phase 3), recording health indicator (**encoder-can't-keep-up done 2026-07-05 via `RecordingHealth` + Degraded UX**; **disk-speed signal done 2026-07-06 via `IDiskSpeedProbe` — an 8 MB write-through probe warns under 10 MB/s, catching network shares/old drives the free-space check alone misses; E2E-verified at 758.5 MB/s on this SSD, correctly silent**), pre-flight checks on every start path (§3.6, Phase 5), capture-source metadata in the library index from day one (**done 2026-07-05: `LibraryIndex` written on finish, Library shows codec·res·duration**), tray quick-record + recent targets (Phase 6), battery warning (**done 2026-07-05: `IPowerStatus` + on-battery pre-flight warning**), first-run encoder benchmark→recommended defaults (Phase 10), DRM-window warning folded into black-frame watchdog. Everything else went to the **grouped §7 backlog** (editing suite, annotation pro, library pro incl. possible SQLite graduation, audio DSP chain, auto-pause, hooks, privacy guard). Mic test = the live meters; low-motion mode noted as mostly inherent to encoders.
- **MVP 1.0-alpha cut (plan §1):** monitor+region, H.264 MP4/MKV, system+mic audio, pause, screenshots, basic library, tray+hotkeys, portable zip. Deferred past MVP: per-app audio, webcam, annotation, scheduler, full AV1/HEVC matrix, MOV/WebM, compact launcher, share polish.
