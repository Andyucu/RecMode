## [Unreleased]

### Fixed
- 2026-07-05 — **Orphan recovery could clobber an in-progress recording (race fix).** Launch-time orphan recovery runs on a background thread while the startup CLI (`--record` / `--tray --record`) may begin a recording — whose crash-safe `*.recording.mkv` temp looks exactly like an orphan. Recovery could grab that live file, remux from a half-written copy, and delete it mid-recording. Recovery now skips any `*.recording.mkv` that's locked for writing (an active recording), only recovering genuine orphans. Verified end to end: a locked temp is left untouched while a free orphan is still recovered to MP4.

### Fixed
- 2026-07-06 — **Eliminated per-frame GPU-view + array allocations in the capture hot path (Phase 2 tail — allocation-free hot-path profiling).** Both `Nv12Converter.Convert` (the recording path, up to 60fps) and `BgraScaler.Scale` (live preview, ~30fps) were creating a **new `ID3D11VideoProcessorInputView` and a new single-element array on every single frame** — a real violation of the plan's §3.9 "allocation-free hot paths" rule, and the concrete cause behind the long-standing "preview is 31%/one-core — headroom to optimize" note. Since WGC's frame pool cycles only 2 physical textures for a capture session's lifetime, the input view is now cached by the texture's native pointer (`ID3D11Texture2D.NativePointer`) and created once per distinct texture instead of once per frame — D3D11 views hold their own internal reference to the resource, so this is safe even though the transient texture wrapper used to create it is disposed each frame. The `[stream]` array literal was replaced with a preallocated field, reused in place. Verified: build clean, 140 tests pass, a real recording (`--selftest-record`) produced a valid 360-frame/6.0s H.264+AAC MP4, and a frame extracted from the middle of the recording plus a live preview screenshot both confirmed clean, uncorrupted GPU output (no artifacts from the view-caching change). Not measured with an external allocation profiler (would need invasive temporary instrumentation in the hot path) — verified by code-level elimination of the exact allocations plus full functional/visual correctness instead.
- 2026-07-06 — **Library header buttons (Videos/Screenshots/Open folder/Refresh) were cramped.** Text sat almost flush against the button edges (user-reported). The design (`RecMode.dc.html`) specifies these as 13px with `6px 12px` padding, but they'd inherited `SecondaryButton`'s 14px default while the view had *shrunk* the button height/padding to 30/`14,0`–`12,0` — too tight for the larger font. Set them to the design's 13px explicitly and restored the standard 32px height, giving the text proper breathing room.

- 2026-07-06 — **Mid-stream hardware→software Degraded fallback (Phase 3 §3.6 tail — closes Phase 3).** When a hardware encoder can't keep up for a sustained period, RecMode now does something about it instead of only warning: past the existing 3 s Degraded threshold, if it's *still* behind 8 s later, the recording rotates to a new segment on a **software** encoder of the same codec (falling back to libx264 as a last resort) — reusing the auto-split rotation machinery, so capture, the state machine, and audio all carry on uninterrupted, and the previous (hardware-encoded) segment is safely finalized first. Only attempted once per recording; software encoders never trigger it (nowhere further to fall back to). The threshold is a new pure, unit-tested `RecordingHealth.ShouldDowngradeToSoftware`. Because this AMD dev box's hardware encoder doesn't realistically fall behind at any resolution tried so far, end-to-end verification uses a new test-only seam (`RecordingCoordinator.TestForceDowngrade`, exercised via a temporary `--selftest-downgrade` hook) that triggers the exact same production rotation path deterministically rather than requiring a genuine overload: verified a hardware `h264_amf` segment rotating cleanly to a `libx264` segment, both valid ffprobe-clean h264/aac MP4s. 3 new tests; 124 tests total; zero warnings.
- 2026-07-06 — **Auto-split large recordings (Phase 3 §3.3 tail).** A new "Auto-split large recordings" toggle (Settings → Recording, with a size combo — 1/2/3.9/8 GB, default ~3.9 GB — safely under FAT32's 4 GB single-file limit) rolls a long recording over to a new file once the current segment crosses the threshold, instead of growing one file indefinitely. Rotation happens seamlessly on the pacer thread: the current segment is finalized (safe-remux to MP4 if enabled, added to the library index) and a new ffmpeg session starts immediately for `Name_part2.ext`, `Name_part3.ext`, … with capture, the state machine, and (if enabled) the audio mixer/pipe all carrying on uninterrupted — the CFR pacer's elapsed-driven catch-up absorbs the brief rotation gap. Segment naming is a new pure, unit-tested `FilenameBuilder.SegmentFileName`. Verified end to end with a forced-low threshold: a ~4.7-minute recording produced two valid segments (107 MB + 6.3 MB, both ffprobe-clean h264/aac MP4s from their own safe-recording MKVs) with two separate library-index entries. 5 new tests; 121 tests total; zero warnings.

- 2026-07-05 — **Scheduler engine — scheduled recordings now actually run (Phase 8).** The Schedule screen and its persisted model landed in Phase 6; this adds the engine that fires them. - 2026-07-05 — **Mid-recording disk-space guard (§3.6).** A long recording that fills the disk previously died abruptly when ffmpeg hit ENOSPC (broken pipe → fatal, risking a truncated finish). The pacer now checks free space on the output drive every ~2 s and, if it drops below a 500 MB critical threshold, **stops the recording gracefully** — finalizing everything captured so far — with a clear "Stopping — the disk is nearly full" warning, rather than crashing. The stop is dispatched off the pacer thread (Stop joins it, so it can't be called inline). The threshold check is a pure, unit-tested `RecordingHealth.IsDiskCritical` (ignores unknown/negative readings). Verified: a normal recording on a disk with plenty free completes with no false positive. Complements the existing under-2 GB pre-flight warning. 96 tests total; zero warnings.
- 2026-07-05 — **Library metadata index (Phase 5 — capture-source metadata "from day one").** Recordings now carry their capture metadata into the Library. On each successful finish the coordinator appends an entry to a `library.json` index — source type (Display/Window/Region), codec, container, resolution, frame rate, and duration — via a new portable-safe, atomic `LibraryIndex` (Core). The Library reads the index and enriches each recording's line to read like "H.264 · 4096×1152 · 0:06 · 567 KB · Today 14:12", falling back to size · date for files with no index entry (older recordings, or ones copied in). Index writes are best-effort and never fail a recording; a corrupt/locked index degrades to the filesystem listing. Verified end to end: a self-test recording wrote an accurate index entry (Display / H264 / Mp4 / 4096×1152 / 60fps / 6s), and 4 new `LibraryIndex` round-trip/replace/persist tests pass. 94 tests total; zero warnings.

- 2026-07-05 — **Recording health indicator (§3.6, promoted 1.0 feature).** RecMode now detects when the encoder can't keep up with real time. The CFR pacer targets `elapsed·fps` frames; if writes back-pressure and the written count falls **more than one second behind** real time (after a 2s grace) and stays there for 3s, the recording is flagged **Degraded**: a one-time DegradedState notice ("The encoder can't keep up — the recording may run slow"), the live stats gain a "⚠ Can't keep up" prefix, and the floating toolbar's rec dot turns amber (as it already does when paused). Health clears automatically when it catches back up. The behind-real-time heuristic lives in a pure, unit-tested `RecordingHealth` (Core) and rides on `RecordingProgress.IsHealthy`. Verified: a normal recording stays healthy (no false positives, valid 360-frame file) and 4 new health unit tests pass. 65 tests total; zero warnings.
