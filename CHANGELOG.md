### Fixed
- 2026-07-07 — **Region source: the picker only ever showed once, then silently reused the same region forever (user-reported).** `RecordViewModel.IsRegionSource`'s setter only opened `RegionSelectWindow` (the picker overlay — which already has both a "Cancel" and "Use region" button, plus Esc/Enter shortcuts) the very first time a region was ever picked; every subsequent switch to the Region tile — across the same session, and across app restarts, since the region persists in settings — silently reused the stored region without ever showing the picker again. ### Fixed
- 2026-07-06 — **Final hardening pass (Phase 10): unhandled exceptions on the recording's background threads could crash the whole app.** `RecordingCoordinator`'s pacer loop (`PaceLoop`, ~60 Hz) only caught `EncoderPipeBrokenException` — any other unexpected exception (a race with capture/session teardown, a null-reference from an edge case, etc.) would propagate off a plain background `Thread`, which .NET treats as a genuinely unhandled exception and **terminates the entire process immediately** (unlike `Task` exceptions, which are non-fatal by default — confirmed this distinction matters here, since several other `Task.Run` call sites in the same file are already safe by virtue of that). Same issue on the audio-pump thread, whose catch only matched `IOException`/`ObjectDisposedException`. Both now catch broadly and tear the recording down gracefully (stop → finalize → report failure) instead of risking a full app crash mid-recording. Also found and fixed two related resource leaks in `SafeTeardown()` (the failed-`Start()` cleanup path): a live `WebcamCaptureSource` (camera device + COM resources) was never stopped, and the audio pump's `CancellationTokenSource` was never disposed — both now cleaned up alongside the existing capture/session/mixer teardown. Verified the refactored pacer loop still behaves correctly for the normal path, pause/resume, and mid-stream segment rotation (three separate self-tests, all passing with the expected frame/segment counts). 154 tests pass; zero warnings.

- 2026-07-06 — **Webcam picture-in-picture overlay (Phase 7).** A new **Webcam** card on the Record screen composites a live camera feed onto the recording (and the live preview) as a corner picture-in-picture box — device picker, corner position (bottom-right/bottom-left/top-right/top-left), and size (10–50% of frame width). Verified without physical camera hardware (none present on this dev machine) via a `--selftest-webcam` hook that injects a synthetic solid-colour frame source in place of a real camera through a test seam (`RecordingCoordinator.TestForceWebcamSource`) — a real recording's extracted frame showed the injected colour landing exactly at the computed bottom-right rectangle. Pure PIP-rectangle geometry (`WebcamOverlayLayout`) is unit-tested (6 tests); the WinRT capture plumbing and GPU compositor are verified via the self-test above rather than mocked. 154 tests total; zero warnings. **Remaining:** verification on real camera hardware (none available here), the pre-existing "Webcam" *source*-type tile (record from the camera as the primary source, distinct from this overlay) stays out of scope — unchanged, still a Phase-1 placeholder.

### Known limitations
- 2026-07-06 — **Per-app audio (process-loopback WASAPI) implemented but disabled — isolation doesn't hold.** Built the full plan-§7 "limit to app" feature: raw COM interop against `ActivateAudioInterfaceAsync`/`AUDCLNT_PROCESS_LOOPBACK_MODE` (not exposed by NAudio), a `ProcessLoopbackCapture : IWaveIn` source, `AudioMixer`/`RecordingCoordinator` plumbing, a running-process picker in the UI, and settings persistence. Interop activation itself works cleanly (no exceptions; confirmed via `Marshal.GetObjectForIUnknown` on the raw `IntPtr` result rather than automatic `UnmanagedType.IUnknown` marshaling, which originally threw `InvalidCastException`/E_NOINTERFACE). But a controlled A/B test proved the capture isn't actually scoped to the target process: with only the target process playing a tone, the meter read a steady 0.0884 RMS; starting a **second, different, non-targeted** process's tone at the same time (target still playing) pushed the meter to 0.0922 — an increase, meaning the second process's audio is leaking into a supposedly single-process capture. Struct layouts (`AUDIOCLIENT_ACTIVATION_PARAMS`, `AUDIOCLIENT_PROCESS_LOOPBACK_PARAMS`, the `PROPVARIANT` blob's field offsets) and GUIDs all check out against recollection of Microsoft's sample, and web-search tools were unavailable to verify further, so the root cause (an interop struct/flag subtlety, most likely something to do with `AUDCLNT_STREAMFLAGS_LOOPBACK` combined with the special `VAD\Process_Loopback` virtual device) is unresolved. Rather than ship a control that implies isolation it doesn't deliver, the "Limit to app" combo is hidden in `RecordView.xaml` and `RecordingCoordinator`/`RecordViewModel` now always pass `null` for the per-app target regardless of what's persisted in `PerAppAudioProcessName` — a real recording today always behaves as full-system audio. All the supporting code (interop file, resolver, mixer parameter, settings field) is left in place, compiling and unit-tested, for a future session to pick up with proper documentation access. 148 tests pass; zero warnings.

### Fixed
- 2026-07-06 — **Eliminated per-frame GPU-view + array allocations in the capture hot path (Phase 2 tail — allocation-free hot-path profiling).** Both `Nv12Converter.Convert` (the recording path, up to 60fps) and `BgraScaler.Scale` (live preview, ~30fps) were creating a **new `ID3D11VideoProcessorInputView` and a new single-element array on every single frame** — a real violation of the plan's §3.9 "allocation-free hot paths" rule, and the concrete cause behind the long-standing "preview is 31%/one-core — headroom to optimize" note. Since WGC's frame pool cycles only 2 physical textures for a capture session's lifetime, the input view is now cached by the texture's native pointer (`ID3D11Texture2D.NativePointer`) and created once per distinct texture instead of once per frame — D3D11 views hold their own internal reference to the resource, so this is safe even though the transient texture wrapper used to create it is disposed each frame. The `[stream]` array literal was replaced with a preallocated field, reused in place. Verified: build clean, 140 tests pass, a real recording (`--selftest-record`) produced a valid 360-frame/6.0s H.264+AAC MP4, and a frame extracted from the middle of the recording plus a live preview screenshot both confirmed clean, uncorrupted GPU output (no artifacts from the view-caching change). Not measured with an external allocation profiler (would need invasive temporary instrumentation in the hot path) — verified by code-level elimination of the exact allocations plus full functional/visual correctness instead.

- 2026-07-06 — **Mid-stream hardware→software Degraded fallback (Phase 3 §3.6 tail — closes Phase 3).** When a hardware encoder can't keep up for a sustained period, RecMode now does something about it instead of only warning: past the existing 3 s Degraded threshold, if it's *still* behind 8 s later, the recording rotates to a new segment on a **software** encoder of the same codec (falling back to libx264 as a last resort) — reusing the auto-split rotation machinery, so capture, the state machine, and audio all carry on uninterrupted, and the previous (hardware-encoded) segment is safely finalized first. Only attempted once per recording; software encoders never trigger it (nowhere further to fall back to). The threshold is a new pure, unit-tested `RecordingHealth.ShouldDowngradeToSoftware`. Because this AMD dev box's hardware encoder doesn't realistically fall behind at any resolution tried so far, end-to-end verification uses a new test-only seam (`RecordingCoordinator.TestForceDowngrade`, exercised via a temporary `--selftest-downgrade` hook) that triggers the exact same production rotation path deterministically rather than requiring a genuine overload: verified a hardware `h264_amf` segment rotating cleanly to a `libx264` segment, both valid ffprobe-clean h264/aac MP4s. 3 new tests; 124 tests total; zero warnings.
- 2026-07-06 — **Auto-split large recordings (Phase 3 §3.3 tail).** A new "Auto-split large recordings" toggle (Settings → Recording, with a size combo — 1/2/3.9/8 GB, default ~3.9 GB — safely under FAT32's 4 GB single-file limit) rolls a long recording over to a new file once the current segment crosses the threshold, instead of growing one file indefinitely.
