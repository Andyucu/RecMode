- 2026-07-05 — **Phase 5 (MVP UX) — part 4: basic Library.** The Library page is now functional (was a stub): a **Videos / Screenshots** tabbed list built directly from the output folders, newest first, with each item showing its name and a `size · date` line (friendly Today/Yesterday/date). Screenshots render a real **thumbnail** (decoded small for the hot path); videos show a play badge. Per-item actions: **Open** (default app), **Show in folder** (Explorer, file selected), and **Delete** (to the **Recycle Bin** via `Microsoft.VisualBasic.FileIO` — a mis-click is recoverable), plus header **Open folder** and **Refresh**. The list loads on navigation and clears on leave (§3.9), and the in-progress `.recording.mkv` temp is filtered out. The richer metadata index (codec/resolution/duration, tags, search) remains a later Library-pro pass. 54 tests pass, zero warnings. Remaining for Phase 5: portable USB acceptance test.
- 2026-07-05 — **Phase 5 (MVP UX) — part 3: countdown + recording toolbar.** A pre-roll **countdown** now runs before an interactive recording start (button/hotkey/tray) when enabled (honours the `CountdownSeconds` setting): a dimmed, full-display overlay ticks N→1 with a subtle per-tick pop, then recording begins; Esc cancels. CLI-initiated recordings (`--record`) skip the countdown — automation means "start now". A **floating recording toolbar** (rec dot, elapsed time, pause/resume, screenshot, live stats, stop) appears bottom-centre for the duration of every recording and hides when it ends, driven by observing the recording state so it covers all stop paths (manual, hotkey, CLI, error). Both overlays are **excluded from the capture** via `SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)` (Win10 2004+/Win11) so they never appear in the recording — verified by comparing WGC captures with and without exclusion (toolbar present vs. absent), plus screenshots confirming both overlays render. The toolbar window is non-activating so it doesn't steal focus from the app being recorded. **Lifecycle change:** the app now runs under `ShutdownMode.OnExplicitShutdown` so transient overlay windows and headless tray operation can't end the process (previously the last window closing quit the app); the main window's close button now quits explicitly, and the `--tray --record`→`--stop` cycle was verified to survive the toolbar opening/closing. 54 tests pass, zero warnings. Remaining for Phase 5: basic Library and portable USB acceptance.
- 2026-07-05 — **Phase 5 (MVP UX) — part 2: CLI + single-instance.** RecMode is now automatable and single-instance. A named-mutex guard makes the first launch the owner; any later launch **forwards its command line** over a named pipe to the running instance and exits, so a second double-click focuses the existing window instead of spawning a duplicate. Recognised flags: **`--record`** (start if idle), **`--stop`** (stop if recording), **`--screenshot`** (capture the current source), **`--tray`** (start headless — the window stays hidden and the app is kept alive by the tray icon). A `--record`/`--screenshot` action loads a default source + encoder on demand (`EnsureDevicesLoaded`) so it works even before the Record view is ever shown (e.g. `--tray --record`). Verified end to end: forwarding a `--screenshot` to a running instance produces a new PNG (~230 ms forwarder round-trip, process count stays at 1); a forwarded `--record` then `--stop` yields a valid ffprobe-checked h264 MP4; `--tray --record` starts recording with no visible window and survives a forwarded stop. 54 tests pass, zero warnings. The temporary `--selftest-*` hooks remain (they bypass the single-instance guard) as the headless verification harness until a dedicated test harness lands. Remaining for Phase 5: countdown overlay, the capture-excluded recording toolbar, basic Library, and portable USB acceptance.
- 2026-07-04 — **Phase 5 (MVP UX) — part 1.** Global hotkeys, tray icon, screenshots, and a failure-mode UX channel. **Global hotkeys** register on a message-only window (F9 start/stop, F10 pause/resume, F11 screenshot); a registration collision with another app surfaces as a RecoverableWarning rather than failing silently (remappable-hotkey UI is Phase 9). **Tray icon** (H.NotifyIcon) with a Show / Start-stop / Screenshot / Quit menu and **minimize-to-tray** — the window hides on minimize while hotkeys keep working headless. **Screenshots**: a one-shot full-resolution WGC grab of the current source (honouring the region crop) saved as PNG to the screenshots folder using the filename pattern and copied to the clipboard, wired to F11, the tray, and a new Screenshot button on the Record screen; verified end to end via a `--selftest-screenshot` hook (valid 5120×1440 PNG). **Failure-mode UX**: a shell snackbar subscribes to the Core `IErrorReporter` — warnings auto-dismiss after 5s, blocking/fatal errors persist until dismissed — giving the §3.6 error taxonomy a real UI channel. Startup wiring verified (app launches with hotkeys + tray live); 54 tests pass, zero warnings. - 2026-07-04 — **Phase 4 (Audio core) — part 1.** Recordings now have sound. New `RecMode.Audio` mixer captures system loopback + microphone (WASAPI), normalizes each to 48 kHz stereo f32 (NAudio resample + mono→stereo), and sums them with per-source gain/mute and a tanh soft-clip, exposing RMS/peak meters. The mix pump is paced to the recording's **active** elapsed time (from the state machine), so audio pauses in lockstep with video and stays synced across pauses. Audio is muxed via a second named pipe into ffmpeg with the codec steered by container — **AAC** for MP4/MOV, **Opus** for MKV/WebM (FLAC where valid). Verified end to end: an A/V self-test produces an MP4 with h264 + AAC (48 kHz stereo) streams whose durations align to ~1 ms (within the ±40 ms target). Added an **Audio mixer card** to the Record screen (System audio + Microphone enable toggles and live meter bars, updated by a ≤30 Hz timer that is torn down on nav-away/minimize per §3.9). 54 unit tests pass (added mixer DSP-math and audio-args suites); zero warnings. Remaining for Phase 4: microphone verification on real hardware, per-source mute/gain controls in the UI, FLAC, caution meter colour, mid-recording mute propagation, and the ±40 ms soak sync test.
- 2026-07-04 — **Phase 3 (Encoding productionized) — part 1.** **Pause/resume with PTS continuity:** the CFR pacer is now driven by the state machine's active elapsed time (writes `Elapsed·fps` frames), and because ffmpeg assigns PTS by frame index at `-r fps`, a paused span simply contributes no frames — the output is gapless with the paused time excluded (verified: a 3s + 2s-pause + 3s session yields exactly 6.0s / 360 frames, not 8s). **Safe recording** (default on): recordings are captured to a crash-safe `.recording.mkv` and remuxed to MP4 with `-c copy` (no re-encode) on stop, then the intermediate is deleted — crash safety and shareability without the user choosing. **Encoder fallback chain** (§3.6): selected → same-codec other backend → any hardware H.264 → libx264, with a typed `EncoderStartException` and a connection timeout so a non-starting encoder can't deadlock the pipe wait. **Disk-space pre-flight guard** (RecoverableWarning under 2 GB free). **Richer live stats** (fps · Mbps · MB) and a Pause/Resume button on the Record screen. 46 tests pass; zero warnings. Remaining for Phase 3: countdown wiring, the full snapshot-tested args matrix, resource-control args (thread cap/effort/priority), auto-split, orphan-MKV detection, mid-stream hardware→software degraded fallback, and the production decision-gate re-check.

- 2026-07-04 — **Phase 0.5 (Pipeline spike) complete — ⚠️ decision gate PASSED (Tier 1).** Disposable spike (`tools/spike/`) proved the full spine end-to-end: WGC monitor capture → D3D11 → GPU BGRA→NV12 (VideoProcessor, single pass) → staging readback → named pipe → ffmpeg → file, CFR-paced, plus a second WASAPI system-audio loopback pipe muxed to AAC. **Validated on AMD (RX 7900 XTX / Ryzen 7 7700X), so the hardware path measured is `h264_amf`** (not NVENC — the plan's dev-machine assumption). **5-minute 1440p60 h264_amf run: 17,992/18,000 CFR frames (0 steady-state drops), 316 MB/s sustained pipe throughput, app CPU 10.2% of one core excluding ffmpeg, memory flat (~108 MB) — all three gate criteria pass.** Encoder/container matrix (h264_amf + libx264 × MP4 + MKV) all produce ffprobe-valid files; native 5120×1440 via libx264 sustained 620 MB/s with 0 stalls (pipeline headroom). Kill-ffmpeg-mid-recording test confirmed the safe-recording design: MP4 dies ("moov atom not found") while a truncated MKV remuxes (`-c copy`) into a clean playable MP4 — recovery proven. Findings: h264_amf caps at 4096 width (ultrawide needs HEVC/AV1 or downscale); a one-time ~95 ms AMF encoder-init stall at t=0; ffmpeg death surfaces as a broken-pipe `IOException` that production must handle (→ `FatalFinalizationError`). Staged the pinned BtbN GPL ffmpeg build in `tools/ffmpeg/` with a committed SHA-256 `ffmpeg.manifest.json`. Added Vortice.Direct3D11/DXGI + NAudio to central package management. **Verdict: no Tier 2 (in-proc hw frames) needed for MVP; NVENC/QSV need their own gate re-check on NVIDIA/Intel hardware.**
