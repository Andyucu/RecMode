### Added
- 2026-07-05 — **Library metadata index (Phase 5 — capture-source metadata "from day one").** Recordings now carry their capture metadata into the Library. On each successful finish the coordinator appends an entry to a `library.json` index — source type (Display/Window/Region), codec, container, resolution, frame rate, and duration — via a new portable-safe, atomic `LibraryIndex` (Core). The Library reads the index and enriches each recording's line to read like "H.264 · 4096×1152 · 0:06 · 567 KB · Today 14:12", falling back to size · date for files with no index entry (older recordings, or ones copied in). Index writes are best-effort and never fail a recording; a corrupt/locked index degrades to the filesystem listing. Verified end to end: a self-test recording wrote an accurate index entry (Display / H264 / Mp4 / 4096×1152 / 60fps / 6s), and 4 new `LibraryIndex` round-trip/replace/persist tests pass. 94 tests total; zero warnings.
- 2026-07-05 — **Remappable global hotkeys (Phase 9).** The three global hotkeys (start/stop, pause/resume, screenshot) are no longer fixed: the Settings → Hotkeys section now has a **Change** button per shortcut that captures the next key chord you press (with Ctrl/Alt/Shift/Win modifiers; Esc cancels), persists it, and re-registers the global hotkey live. Chords round-trip through a new pure, unit-tested `HotkeyChord` (Core) — "Ctrl+Shift+F9" ⇄ modifier mask + virtual key — and `HotkeyBindings` now reads the chords from settings (falling back to F9/F10/F11 if a stored value is blank or unparseable) with a `Rebind()` for live re-registration. Verified: 15 new `HotkeyChord` tests (parse/format/round-trip/ordering); launching with custom chords (Ctrl+Alt+R, Shift+F8) registers them cleanly with no collisions, and an invalid stored chord falls back to the default without crashing. 90 tests total; zero warnings. (The interactive capture UI builds and is wired through the same tested chord code; its on-screen render wasn't screenshot-verified this run because the app window opened behind the IDE and couldn't be foregrounded programmatically.)
- 2026-07-05 — **FLAC audio verified end to end (Phase 4).** Confirmed the lossless-audio path works through the real pipeline: with MKV + FLAC configured, a CLI-driven recording produced an ffprobe-valid file with a **flac** audio stream (48 kHz, stereo) alongside h264 video. FLAC is steered to MKV only (MP4/MOV fall back to AAC, WebM to Opus, as designed). Locked the FLAC-on-MKV command line with a snapshot test (encoding tests 31→32, total 75). No code change was needed — the path was already correct; this closes the Phase 4 FLAC tail with evidence.
- 2026-07-05 — **On-battery pre-flight warning (§3.6 / Phase 9, promoted 1.0 feature).** Starting a recording while on battery now surfaces a RecoverableWarning ("You're recording on battery power (NN% left) — plug in for long sessions"), since capture + encode are power-hungry. Power state is read through a new `IPowerStatus` abstraction over Win32 `GetSystemPowerStatus` (so the decision is mockable), and included in the recording pre-flight. Verified: on an AC desktop with no battery it correctly stays silent (no false positive) and recording proceeds normally. 74 tests pass, zero warnings.
- 2026-07-05 — **Encoder effort tiers (Phase 3 §3.3).** A **Fast / Balanced / Quality** effort setting now trades encode speed against compression, mapped per encoder to its native preset — libx264/libx265 `ultrafast/veryfast/medium`, SVT-AV1 `10/8/6`, NVENC `p2/p4/p6`, AMF `speed/balanced/quality`, QSV `veryfast/medium/slow`. **Balanced deliberately keeps each encoder's existing default**, so nothing changes unless you pick another tier (and the snapshot tests stay green). Exposed as an "Encoder effort" combo in the Settings → Performance section, flowing from settings through the `FfmpegJob`. Verified: the card renders and defaults to Balanced; nine new arg-mapping tests cover x264/NVENC/AMF across all three tiers. 74 tests total; zero warnings.
- 2026-07-05 — **Recording health indicator (§3.6, promoted 1.0 feature).** RecMode now detects when the encoder can't keep up with real time. The CFR pacer targets `elapsed·fps` frames; if writes back-pressure and the written count falls **more than one second behind** real time (after a 2s grace) and stays there for 3s, the recording is flagged **Degraded**: a one-time DegradedState notice ("The encoder can't keep up — the recording may run slow"), the live stats gain a "⚠ Can't keep up" prefix, and the floating toolbar's rec dot turns amber (as it already does when paused). Health clears automatically when it catches back up. The behind-real-time heuristic lives in a pure, unit-tested `RecordingHealth` (Core) and rides on `RecordingProgress.IsHealthy`. Verified: a normal recording stays healthy (no false positives, valid 360-frame file) and 4 new health unit tests pass. 65 tests total; zero warnings.
- 2026-07-05 — **Snapshot-tested ffmpeg args matrix (Phase 3 §3.3).** Added six snapshot tests that pin the **exact** ffmpeg command line for a representative matrix — libx264/MP4 and /MKV (faststart on vs off), h264_amf/MKV, libx264/MP4 with AAC audio, SVT-AV1/WebM with Opus audio, and libx264/MP4 with a thread cap — so a quarterly ffmpeg re-pin or a careless edit can't silently change the encode without a failing test. All pass (whitespace-normalised full-string equality). Encoding tests 16→22, total 61; zero warnings.
- 2026-07-05 — **Orphaned-recording recovery on launch (Phase 3 crash safety).** A session that crashes mid-recording leaves a crash-safe `*.recording.mkv` behind (the safe-recording design); RecMode now **recovers these on the next launch**. A new `OrphanRecoveryService` scans the recordings folder at startup (off the UI thread) and remuxes each orphan to a playable MP4 with `-c copy` (no re-encode, avoiding clobbering an existing file), removes the temp, and surfaces a one-line "recovered a recording from a previous session" notice. The remux logic was extracted into a shared `Remuxer` used by both this and the safe-recording stop path. Verified end to end: a valid `*.recording.mkv` placed in the recordings folder was remuxed to a valid ffprobe-checked H.264 MP4 and the temp deleted on the next launch. 55 tests pass, zero warnings.
- 2026-07-05 — **Per-source audio volume (Phase 4 mixer UI / Record fidelity).** The Record screen's audio mixer now has a **volume slider** (0–100%, with a live percentage) under each source's enable toggle and meter, matching the design's per-source mixer. Volume maps to mixer gain and takes effect everywhere: on the live meters, on the recording at start, and **mid-recording** — a new `RecordingCoordinator.SetAudioGains` propagates changes to the in-progress recording mixer. Volumes persist per source (`SystemVolume`/`MicVolume`). Sliders disable when their source is off. Verified on the real GUI (UI-Automation): the sliders render with % labels, and dragging the system volume to 60 persisted `SystemVolume=60`. Resolves the Phase 4 per-source-gain-UI and mid-recording-propagation tails. 55 tests pass, zero warnings.
- 2026-07-05 — **Encoder resource controls (Phase 3 §3.3 / Phase 9 Performance group).** Recording now honours the resource-efficiency levers that were modelled but unwired: a **CPU thread cap** for software encoders (`-threads N`, emitted only for software — hardware offloads to the GPU) and a **below-normal ffmpeg process priority** so recording never starves foreground work. Both flow from settings through the `FfmpegJob` into the args/process, and a new **Performance** section in Settings exposes them (thread-cap combo with an "Auto" = all-cores option, and a "Lower encoder process priority" toggle), each persisting immediately. Verified: a new `-threads` args test (software-only, absent for hardware and for Auto); the Performance section renders with correct icons and the priority toggle persists to `settings.json`. 55 tests pass (was 54), zero warnings.
